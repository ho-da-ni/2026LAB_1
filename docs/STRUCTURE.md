# 기본 폴더 구조와 설계 이유

## 권장 구조

```text
LAB/
├─ README.md
├─ docs/
├─ src/lab/
├─ tests/
├─ examples/
└─ artifacts/
```

## 디렉터리별 역할

### `README.md`
프로젝트의 가장 짧고 강한 기준 문장과 핵심 원칙을 담는다.
새로운 참여자는 먼저 이 파일을 읽고 프로젝트가 무엇을 하는지, 무엇을 하지 않는지 이해해야 한다.

### `docs/`
실제로 생성하거나 생성해야 하는 문서의 기준 포맷을 둔다.
문서 구조를 먼저 고정하면, 이후 IR 설계와 CLI 출력 포맷을 안정적으로 맞출 수 있다.

### `src/lab/`
Python 패키지 본체를 둔다.
CLI 진입점, 정적 분석 로직, IR 생성 로직, 렌더러를 이 하위에서 역할별로 확장하기 좋다.

### `tests/`
결정론성, fingerprint, 렌더링 결과 비교 같은 검증을 자동화한다.
이 프로젝트는 "맞는 결과를 만들기"뿐 아니라 "항상 같은 결과를 만들기"가 중요하므로 테스트 디렉터리가 필수다.

### `examples/`
샘플 입력과 기대 산출물을 둔다.
실제 개발 중에 예제를 회귀 테스트처럼 사용할 수 있어 설명 가능성과 재현 가능성이 높아진다.

### `artifacts/`
실행 결과 예시, 템플릿 산출물, 비교용 출력 파일을 저장한다.
소스와 생성물을 분리하면 변경 추적과 디버깅이 쉬워진다.

## 왜 이 구조가 LAB에 적합한가

1. **입력, 구현, 출력, 검증을 분리할 수 있다.**
   - `examples/`는 입력
   - `src/`는 구현
   - `artifacts/`와 `docs/`는 출력
   - `tests/`는 검증

2. **결정론성과 재현 가능성에 유리하다.**
   각 책임이 섞이지 않으면 어떤 입력에서 어떤 출력이 나왔는지 추적하기 쉽다.

3. **문서 중심 개발이 가능하다.**
   LAB는 문서 산출물이 핵심이므로 `docs/`가 먼저 있어야 구현도 문서 형식에 맞춰 설계할 수 있다.

4. **초기에는 단순하고, 이후 확장도 쉽다.**
   작은 저장소에서는 과하지 않으면서도, 추후 `src/lab/analyzers`, `src/lab/renderers`, `src/lab/ir` 같은 세분화가 자연스럽다.
